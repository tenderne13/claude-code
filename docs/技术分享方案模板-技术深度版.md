- 工程限制：`[研发周期 / 历史包袱 / 多团队协作 / 成本约束]`

### 2.2 技术问题定义
`[明确本次分享真正要解决的问题，避免主题过泛]`

示例：
- `[某框架]` 在高并发场景下的性能瓶颈在哪里

`[说明问题的普适性、复杂度和实际价值]`

示例：

- 输出一份可落地的实践建议

`[写清核心听众是谁，他们现在卡在哪一层]`

示例：

- 遇到问题只能靠经验试错

### 5.2 预期收益

## 6. 核心内容设计

### 6.1 建议内容结构
1. 问题背景与典型现象
2. 技术原理与关键设计
3. 核心链路拆解
4. 实际案例分析
5. 优化方案与权衡
6. 最佳实践与适用边界

### 6.2 分章节模板

#### 第一部分：问题现象与分析入口
**目标**  
`[先让听众看到问题，而不是直接讲结论]`

**重点内容**
- `[现象描述]`
- `[指标表现 / 日志特征 / 报错模式]`
- `[为什么这个现象值得深挖]`

**讲解建议**
- 用真实案例开场
- 用数据或图表建立问题紧迫感

#### 第二部分：核心原理
**目标**  
`[解释系统为什么会这样工作]`

**重点内容**
- `[核心模型]`
- `[核心流程]`
- `[关键数据结构 / 算法 / 状态机 / 协议]`

**讲解建议**
- 尽量画图，不要只堆概念
- 明确输入、处理、输出三个环节

#### 第三部分：关键实现拆解
**目标**  
`[把抽象原理落到具体实现]`

**重点内容**
- `[主链路时序]`
- `[关键模块职责]`
- `[核心配置项 / 参数 / 阈值]`
- `[异常路径 / 边界条件]`

**讲解建议**
- 使用流程图、时序图、模块图
- 必要时贴少量关键代码，避免大段源码

#### 第四部分：案例与复盘
**目标**  
`[说明原理如何映射到真实问题]`

**重点内容**
- `[线上案例背景]`
- `[问题定位过程]`
- `[最终根因]`
- `[修复措施]`
- `[改进效果]`

**讲解建议**
- 强调“如何定位”的过程
- 展示错误假设如何被排除

#### 第五部分：优化与权衡
**目标**  
`[让听众理解没有银弹，方案都有前提]`

**重点内容**
- `[优化策略 1]`
- `[优化策略 2]`
- `[不同方案对比]`
- `[收益、成本、适用条件]`

**讲解建议**
- 使用对比表
- 强调 trade-off，而不是只给单一答案

## 8. 图示与材料设计

### 8.1 建议图示
- 架构图：说明整体模块关系
- 时序图：说明关键链路调用顺序
- 数据流图：说明数据如何流转与转换
- 状态图：说明状态变化和边界条件
- 对比表：说明不同方案差异

### 8.2 建议材料
- PPT
- 核心流程图
- 关键代码片段
- 压测数据 / 线上监控截图
- 故障案例时间线
- FAQ 文档

### 8.3 代码展示原则
- 只展示关键片段，不展示大段实现
- 每段代码回答一个明确问题
- 代码要能支撑结论，而不是增加认知负担

## 9. 时间规划建议

| 环节 | 内容 | 建议时长 |
|---|---|---|
| 开场 | 背景、问题定义、目标 | `5~10 分钟` |
| 原理部分 | 核心机制讲解 | `15~20 分钟` |
| 实现拆解 | 链路、模块、配置分析 | `15~20 分钟` |
| 案例复盘 | 故障、定位、修复、效果 | `15~20 分钟` |
| 总结 | 结论、建议、边界 | `5~10 分钟` |
| Q&A | 互动答疑 | `10~15 分钟` |

## 11. 风险与应对

### 11.1 常见风险
- 内容过深，超出多数听众知识边界
- 原理讲太多，缺少案例支撑
- 案例太碎，无法抽象出通用方法
- Demo 或代码细节过多，挤占主线内容

### 11.2 应对策略
- 先讲问题再讲原理，降低抽象门槛
- 每一部分都回到真实场景
- 控制代码和源码细节比例
- 用结论页反复收束主题
